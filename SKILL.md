# fitness-coach-lite

Telegram 低交互场景的个人运动追踪教练。把"像发消息一样记录运动"作为核心目标。

---

## 核心哲学

**3秒记录原则**：用户在任何对话轮次里，发来一条训练或状态消息，agent 应当能在最少回复轮次内完成记录。

---

## 数据文件（统一在 data/ 下）

```
data/
├── plan.json              # 当前训练计划
├── history.json           # 统一历史：训练记录+状态记录+快照引用
├── body_status.json       # 身体基本信息+体重/体脂历史
└── .session.json          # 训练计时会话（临时，训练结束后自动删除）
```

**注意：没有 `.current_workout.json`**（已废弃，旧版遗留）。

## 触发判断（每条用户消息都要判断）

当用户发送消息时，按以下顺序判断意图：

### 1. 训练记录类（最高优先级）
关键词/模式：`跑了`、`练了`、`深蹲`、`卧推`、`走了多少步`、`拉伸`、`骑车`、`上半身`、`今天训练了`、`做了.*组`、`Xkg.*X次`、`开始练`、`练完了`、`第.*组`

**⚠️ 硬性规则：记录前必须先查当前状态**

每次用户报告训练成绩之前，**必须**先调用 `workout.py status` 确认今日已完成的动作和组数，明确知道是「第几组」再记录。

**处理流程：**
1. 调用 `workout.py status` 查看今日已完成的内容
2. 结合用户描述，判断是「追加」还是「新做某组」
3. 调用 `workout.py log "<date>" '<json>'` 写入 history.json
4. 记录后再次调用 `workout.py status` 确认

**`log` 命令格式（追加模式）：**
```bash
# 单个动作（自动包装成 exercises 列表）
python3 workout.py log "2026-04-25" '{"name":"坐姿划船","weight_kg":25,"sets":[{"reps":8},{"reps":8},{"reps":8}]}'

# 多个动作
python3 workout.py log "2026-04-25" '{"exercises":[{"name":"坐姿划船","weight_kg":25,"sets":[{"reps":8},{"reps":8}]},{"name":"高位下拉","weight_kg":25,"sets":[{"reps":8}]}]}'
```

**追加行为（关键）：**
- 同一动作多次调用 → **合并 sets**（不覆盖，只追加组数）
- 不同动作调用 → **分别追加**
- 每天只有一条 workout 记录，所有动作在里面合并

**⚠️ 重要：向 agent 报告时的明确格式**
- 说「第3组10个」→ agent 理解为追加第3组
- 说「第1组10个，其余还是8个」→ agent 理解为修正第1组（或直接说「第1组改成10个」）
- 说「Chest press 10个」→ agent 追问「是第1组还是又做了一个？」

为了避免歧义，**建议用户报告时用组号**：「第1组 10个」「第2组 8个」

**格式灵活接受：**
- `今天跑了40分钟` → `{summary:"跑步40分钟"}`
- `深蹲60kg 5x5，卧推30kg 5x5` → 两个动作分别记录
- `晚上练了腿` → `{summary:"练腿"}`
- `走了8000步` → `{summary:"步行8000步"}`
- `今天拉伸了15分钟` → `{summary:"拉伸15分钟"}`
- `上半身训练` → `{summary:"上半身训练"}`

**推荐工作流（可选，不是必须的）：**
1. 用户说「开始练」→「开始训练」→ 调用 `workout.py start-session`（记录开始时间到 `.session.json`）
2. 用户报每组成绩如「第1组 10个」「第二组 8个」→ 调用 `workout.py log "<date>" '<json>'` 实时追加
3. 用户说「练完了」→ 调用 `workout.py finish-workout`（从 `.session.json` 读取开始时间，计算时长）

> ⚠️ `start-session` + `finish-workout` 是**可选的计时流程**，不调用也可以直接用 `log` 记录。

### 2. 状态记录类
关键词/模式：`累`、`没睡好`、`不想练.*`、`这周只能.*次`、`出差`、`经期`、`恢复`、`不舒服`、`膝盖.*`、`状态`

**处理流程：**
1. 提取状态类型和详情
2. 调用 `workout.py status-log "<date>" <status> <detail>` 写入 history.json

**状态类型映射：**
| 用户词 | 存储值 |
|--------|--------|
| 累、很累、疲惫 | `tired` |
| 没睡好、睡眠差 | `poor_sleep` |
| 出差 | `travel` |
| 经期、生理期 | `period` |
| 不想练X、讨厌练X | `avoiding:{部位}` |
| 这周只能练X次 | `frequency_limit:{N}` |
| 恢复、休息一下 | `recovery` |
| 膝盖不舒服、肩膀疼 | `injury:{部位}` |
| 状态不错 | `good` |
| 正常 | `normal` |

### 3. 查询报告类
关键词/模式：`今天练了什么`、`这周练了几次`、`报告`、`周报`、`月报`、`这周为什么执行.*`、`最近计划改了.*`、`练什么`

**处理流程：**
1. **先判断今天周几**：调用 `workout.py plan view`，内部已根据 `date.today()` 自动确定真实日期对应的星期
2. **再查今日计划**：`workout.py plan view` 输出中已包含今日训练内容（标注了当天）
3. **其他查询**：
   - `今天练了什么` → 查 history.json 中今日 type=workout 记录
   - `这周练了几次` → 统计本周 type=workout 数量
   - `这周状态` → 查本周 type=status 记录
   - `日报` → 调用 `workout.py report today`
   - `周报` → 调用 `workout.py report week`
   - `月报` → 调用 `workout.py report month`
   - `阶段总结` → 调用 `workout.py report summary`

### 4. 计划管理类（优化版）
关键词/模式：`新计划`、`改训练计划`、`更新计划`、`训练计划.*调整`、`分析训练`、`计划建议`、`调整计划`、`更新建议`、`优化训练计划`

**❗ 第一步：先判断用户每个大部位的当前阶段（不是统一的全局阶段）**

同一用户不同部位可能处于不同阶段，必须逐个分析：

| 大部位 | 重建期特征 | 稳定发力期特征 | 进阶期特征 |
|--------|-----------|---------------|-----------|
| **背** | 高位下拉第6个代偿、划船肩代偿 | 能标准完成8-12次无代偿 | 重量稳步增长、可分化训练 |
| **胸** | Chest press第6组肩酸、发力感差 | 平板卧推稳定、能真正力竭 | 可加变体动作（上斜/哑铃） |
| **肩** | 推类就带动肩、前锯肌控制差 | 侧平举稳定、肩胛稳定 | 可独立肩日 |
| **腿** | Leg press晃、腰代偿 | 深蹲/腿压稳、能大重量 | 可加单腿动作 |
| **核心** | 卷腹颈椎发力、平板不稳 | 核心控制良好 | 可负重核心训练 |

**判断方法（从 history.json + 用户反馈提取）：**

| 信号 | 指向阶段 |
|------|---------|
| 「第6个开始代偿」「肩膀酸」「发力找不到」 | 重建期（该部位） |
| 「能做完8-12个」「没有不舒服」 | 稳定发力期（该部位） |
| 「太轻了」「做完很轻松」「想加重量」 | 进阶期（该部位） |

**输出格式：**
```
**🔍 各部位阶段判断**
| 部位 | 当前阶段 | 依据 |
|------|---------|------|
| 背 | 稳定发力期 | 高位下拉25kg×5×8无代偿，划船稳定 |
| 胸 | 发力重建期 | 第6组开始肩酸，未真正力竭 |
| 腿 | 进阶期 | Leg press 45kg×4×10很稳 |
| 核心 | 稳定发力期 | 卷腹35kg×3×10无问题 |
```

**计划策略（按部位分配）：**

- **重建期部位** → 激活组(先) → 主项 → 校准组(后)；不加孤立/容量动作；主项刺激要足够（Chest press 必须有1-2组接近力竭）
- **稳定发力期部位** → 可加1-2个相关动作，组数3-4组
- **进阶期部位** → 可分化训练、加变体、追重量

**❗ 重建期胸部主项刺激标准：**
- Chest press 必须有至少1组"明显变难"（接近力竭但不代偿）
- 推荐方案（二选一）：
  - **方案A（推荐）**：前3组正常重量×8，第4组做递减或暂停休息后继续至接近力竭
  - **方案B（安全）**：4组 8/8/7/6，允许自然掉次数，最后一组明显变难

**⚡ 激活组标准（有训练价值的激活）：**
- 重量：有控制的轻微疲劳（不是自重"动一下"）
- 次数：8-12次
- 组数：2-3组
- 目的：让目标肌群真的变强，而不只是唤醒

**🔧 校准组标准：**
- 重量：有控制的轻微疲劳
- 次数：8-12次
- 组数：2-3组
- Face pull 是背日校准核心动作，必须有训练价值（15kg×10-12×2-3）

**💪 主动作标准：**
- 不在组间穿插其他动作
- 组间不打断发力连续性
- 最后一个动作的最后一组做完才结束该日训练

**⚠️ 重要：同一个训练日可能包含不同阶段的部位**
- 周一背（稳定期）：标准主项 + 可加容量动作
- 周二胸（重建期）：轻量主项 + 修复动作
- 周四腿（进阶期）：可加重 + 加组数

**❗ 第二步：分类每个动作的真实作用（按当前阶段，不是标准健身房逻辑）**

| 类型 | 作用 | 安排方式 | 示例 | 重量/次数标准 |
|------|------|----------|------|---------------|
| **⚡ 激活组** | 唤醒目标肌群、启动神经通路 | 在主动作**之前** | 悬挂肩胛下沉、极轻飞鸟 | **有控制的轻微疲劳**，8-12次，2-3组 |
| **🔧 校准组** | 修复/强化薄弱环节，重新校准发力位置 | 在主动作**之后** | Face pull、悬挂肩胛下沉 | **有控制的轻微疲劳**，8-12次，2-3组 |
| **💪 主项动作** | 真正的训练负载，真正刺激目标肌群 | 每天1-2个，每个3-4组，整组做完才换下一个动作 | 高位下拉、Chest press、Leg press、坐姿划船 | 标准重量，3-4×8-12 |
| **🌊 容量/辅助动作** | 主项稳定后才加，增加训练量 | 重建期一律不加 | 哑铃弯举、绳索下压、侧平举 | — |

**⚡ 激活组标准（有训练价值的激活）：**
- 重量：有控制的轻微疲劳（不是自重"动一下"）
- 次数：8-12次
- 组数：2-3组
- 目的：让目标肌群真的变强，而不只是唤醒

**🔧 校准组标准：**
- 重量：有控制的轻微疲劳
- 次数：8-12次
- 组数：2-3组
- Face pull 是背日校准核心动作，必须有训练价值（15kg×10-12×2-3）

**💪 主动作标准：**
- 不在组间穿插其他动作
- 组间不打断发力连续性
- 最后一个动作的最后一组做完才结束该日训练

**关键动作的当前意义（最容易误判）：**

| 动作 | ❌ 标准健身房逻辑 | ✅ 当前实际意义 |
|------|-----------------|----------------|
| Face pull | 肩后束孤立 → 归入肩日 | **背校准动作** → 先做背主项，**后做**Face pull 校准肩胛稳定肌（15kg×10-12，有训练价值） |
| 飞鸟 | 胸孤立 → 归入胸日 | **胸激活组** → 先极轻飞鸟×10（激活胸），再做Chest press主项 |
| 飞鸟 | 胸孤立 → 归入胸日 | **胸激活组** → 先极轻飞鸟×10（激活胸），再做Chest press主项 |
| 反向飞鸟 | 肩后束孤立 → 归入肩日 | **肩胛校准** → Face pull同类，放在主项后 |
| 哑铃弯举 | 二头孤立 → 背日加 | **重建期暂不加** → 专注主项 |
| 绳索下压 | 三头孤立 → 胸日加 | **重建期暂不加** → 主项没稳定不加小肌群 |
| 侧平举 | 肩中束孤立 → 独立肩日 | **重建期暂不加** → Face pull代替；周末可试 |
| 上斜卧推 | 胸变体 → 胸日主项 | **重建期暂不加** → 先把平板Chest press稳定；周末可试 |

**背训练日标准顺序（激活→主项→校准）：**
```
1. ⚡ 激活组：悬挂肩胛下沉 — 自重 × 2×10（激活下斜方）
2. 💪 主项：高位下拉 — 25kg × 4×8
3. 💪 主项：坐姿划船 — 25kg × 3×8
4. 🔧 校准组：Face pull — 15kg × 10-12 × 2（校准肩胛稳定肌，有训练价值）
```
> ⚠️ Face pull **不是第一个动作（不是激活）**，而是**最后一个动作（校准）**。必须用 15kg×10-12×2-3，有控制的轻微疲劳，不能只是"动一下"。自重×3×5 几乎没有训练价值。

**按部位分配计划（不是统一模板）：**

同一个训练日可能包含不同阶段的部位：
- 某部位在重建期 → 该日限制组数、加修复动作、不加孤立
- 某部位在稳定期 → 可正常安排主项+1-2个关联动作
- 某部位在进阶期 → 可加重量、加班数、加变体

**重建期核心原则（适用于该部位当前处于重建期时）：**
1. **激活组先于主项** — 唤醒目标肌群，建立神经通路，再做主项
2. **校准组在主项之后** — 修复/强化薄弱环节，重新校准发力
3. **激活和校准必须有训练价值** — 有控制的轻微疲劳，不能只是"动一下"（如Face pull必须15kg×10-12，不能自重×3×5）
4. **主动作之间不打断发力连续性** — 整组做完才换下一个动作，不在组间穿插
5. **小肌群暂不分离** — 该部位重建期时，不单独加二头/三头/侧平举
6. **训练时间匹配现实** — 工作日10-20分钟 → 每天1-2个主项+激活/校准

**❗ 第三步：计算实际训练时长（必须）**

在生成计划前，**必须**从 history.json 统计近期实际训练时长：

```python
# 从 history.json 提取每条 workout 的 duration_min
# 计算近14天平均训练时长
```

| 时长区间 | 建议 |
|---------|------|
| < 15分钟 | 工作日极短，建议每天1个主项 + 修复动作 |
| 15-25分钟 | 每天1-2个主项 + 修复动作，适合重建期 |
| 25-40分钟 | 可安排2个主项 + 修复动作 |
| > 40分钟 | 可安排较完整的训练结构 |

**输出格式：**
```
⏱️ **近期实际训练时长分析**
- 近14天训练 N 次
- 平均时长：约 N 分钟/次
- 范围：N-N 分钟
- 建议每天训练时长：N-N 分钟
```

**❗ 必须询问用户确认后再生成计划：**
> 「根据你过去的训练数据，平均每天练 N 分钟。你希望每天还是这个时长，还是调整一下？」

收到用户回复后再进入 Step 4（生成建议）。

## 训练建议动态调整

### 触发条件
以下状态变化后，agent 应主动或被动更新训练建议：
- 用户说`出差`、`经期`、`恢复模式`、`这周只能练X次` → 降频建议
- 用户说`不想练腿`、`讨厌练背` → 调整部位偏好
- 用户连续3天状态为`tired` → 建议降强度
- 用户连续`poor_sleep` → 建议推迟主训练日

### 调整输出格式
当触发调整时，输出格式：
```
📋 训练建议已调整（v{N}）
原因：{触发原因}
建议：{具体建议}
```

### 方案版本记录
每次调整训练建议，调用 `workout.py plan version <reason>` 记录到 history.json 的 plan_versions 数组。

## Telegram 低摩擦输入设计原则

1. **不需要确认轮次**：记录完成后直接显示"✅ 已记录：..."
2. **模糊重量可接受**：若用户说"练了腿"无细节，仍可记录为 `{summary:"练腿"}`
3. **同一条消息可混合记录**：如"今天跑了30分钟，状态很累" → 一次性写入训练+状态
4. **查询即时响应**：不要求二次确认，直接查直接回
5. **周报/月报一次性输出**：不分段确认

## ⚠️ 技术注意事项

### 日期使用系统 UTC+8 时钟

`workout.py` 脚本直接使用系统 UTC+8 时钟（`datetime.now(timezone(timedelta(hours=8)))`），不需要任何环境变量。

### `.session.json` vs `.current_workout.json`

- **`.session.json`**：由 `start-session` 创建，`finish-workout` 删除。存今日计划+开始时间。
- **`.current_workout.json`**：由 `start` 命令（新版step流程）创建。**已废弃，不建议使用**。
- `log` 命令**不依赖**任何临时文件，直接读写 history.json。

### 历史教训：duration_min 显示为0的 bug

**现象**：`report today` 显示「⏱️ 约 0 分钟」，即使真实训练已过几分钟。

**根因**：`_history_add` 中 `duration_min` 用 `int((logged_at - session_started_at).total_seconds() / 60)` 计算，<60秒结果为0（向下取整）。`_report` 的 `if dur:` 把 `dur=0` 当 falsy 跳过时间行。

**修复方向**：`report` 中 `<1分钟` 显示分支要先于 `if dur:` 检查；`history.json` 写 `duration_min: 0` 后后续追加不更新（updated判断失效）。**重要**：debug 时不要 `reset-today` 清除真实数据。

### 调试经验

- `reset-today` 会清空 `history.json` 中的今日训练数据，调试前务必确认是否需要保留真实记录
- `terminal` 命令默认 60 秒超时，长 `sleep` 测试会失败；用 `process` 后台执行可以避免
- `history.json` 调试时用 `cat` 查原始数据，比 `report` 输出更可靠
- `log` 追加模式：同动作名→合并sets；不同动作→分别追加；每天只有一条 workout 记录
